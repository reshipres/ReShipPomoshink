# Уроки из клиентских переписок

Источник: локальный Telegram export `DataExport_2026-06-11.zip`. Сырой текст, имена, телефоны, адреса и ссылки в репозиторий не сохраняются.

## Что важно для качества бота

- Бот должен понимать короткие подтверждения: `да`, `нет`, `ага`, `окей`, `хорошо`, `понял`, `спасибо`, `отлично`. Иначе он слишком часто отвечает общим fallback-ом.
- Клиенты часто отправляют телефон, ФИО, ПВЗ, город или адрес отдельным сообщением. Это не обычный вопрос: такие сообщения надо воспринимать как данные к заказу и передавать оператору, если нет безопасного автоматического действия.
- Проблемы с сайтом, корзиной, оформлением и промокодом являются отдельным support-сценарием. Бот не должен отвечать общими правилами оплаты, если клиент не может оформить заказ.
- Внешние ссылки, Poizon/Taobao/Dewu/1688 и просьбы "заказать по ссылке" требуют ручного расчета. Бот должен быстро передавать такие запросы оператору и просить размер/цвет/количество одним сообщением.
- Вопросы "сколько ждать", "как долго", "сроки", "когда будет" часто относятся не к текущему заказу, а к доставке, предзаказу или поставке. Нужно отличать их от "где мой заказ".
- Негативные сообщения про задержку, деньги, ошибку или отсутствие ответа лучше эскалировать раньше, чем пытаться угадывать шаблонный ответ.

## Как проверять новые правки

```bash
npm run analyze:telegram -- /Users/davidbukarov/Downloads/DataExport_2026-06-11.zip
npm test
npm run eval
```

Текущий прогон по этому архиву после первого пакета правил:

- входящих текстовых сообщений: 28 652;
- `other`: 9 929 сообщений;
- `otherRate`: 34.65%.

`analyze:telegram` по умолчанию печатает только агрегаты. Для локальной отладки можно включить редактированные примеры:

```bash
npm run analyze:telegram -- /Users/davidbukarov/Downloads/DataExport_2026-06-11.zip --examples
```
